## WebLLM이란?

[WebLLM](https://webllm.mlc.ai/)은 **브라우저 안에서 LLM(대규모 언어 모델)을 실행**하는 추론 엔진이다.

일반적인 ChatGPT/Claude 같은 AI 서비스는 서버에서 모델을 실행하고 네트워크로 결과를 보내주지만,
WebLLM은 **서버 없이 사용자의 브라우저(클라이언트)에서 직접 모델을 로드하고 추론**한다.

```
일반 LLM 서비스:
  사용자 브라우저 ──── 인터넷 ────► GPU 서버 (모델 실행) ──► 응답 반환

WebLLM:
  사용자 브라우저 (WebGPU로 모델 직접 실행) ──► 즉시 응답
  서버 불필요, 네트워크 불필요
```

**핵심 기술**: WebGPU를 활용한 GPU 가속 추론
**개발**: MLC (Machine Learning Compilation) 팀
**npm 패키지**: `@mlc-ai/web-llm`

## 어떻게 동작하는가?

### 동작 흐름

```
1. 브라우저가 모델 파일 다운로드 (최초 1회, 이후 캐시)
   └─ 모델 가중치 (.wasm + .bin 파일) → IndexedDB에 저장

2. WebGPU로 GPU 초기화
   └─ navigator.gpu.requestAdapter() → requestDevice()

3. WASM 런타임이 모델을 GPU 메모리에 로드
   └─ MLC 컴파일된 모델 → WebGPU Compute Shader로 실행

4. 사용자 입력 → 토크나이저 → GPU 추론 → 디토크나이저 → 텍스트 출력
   └─ OpenAI 호환 API (chat.completions.create)
```

### 기술 스택

| 계층 | 기술 | 역할 |
|------|------|------|
| **API** | OpenAI 호환 인터페이스 | `chat.completions.create()` — 기존 코드 재사용 가능 |
| **런타임** | TVM (Apache TVM) | ML 컴파일러 — 모델을 WebGPU용으로 최적화 |
| **실행** | WebGPU + WASM | GPU 가속 추론 + WASM으로 전처리/후처리 |
| **저장** | IndexedDB | 모델 파일 캐시 (재방문 시 다시 다운로드 안 함) |
| **스레딩** | Web Worker | 추론을 별도 스레드에서 실행 → UI 블로킹 방지 |

```

├── index.html          ← 채팅 UI (다크 테마)

├── tsconfig.json
├── docs/
│   └── webllm-overview.md  ← 이 문서
└── src/
    ├── main.ts         ← 엔진 로드 + 채팅 로직 + 스트리밍
    └── worker.ts       ← Web Worker (UI 블로킹 방지)
```

## 실행 방법

```bash
cd webllm
npm install
npm run dev
# → http://localhost:5400 에서 열기 (Chrome, WebGPU 지원 필요)
```

**요구사항**: Chrome 113+ 또는 Edge 113+ (WebGPU 지원)
**첫 실행**: 모델 다운로드에 시간이 걸림 (~1.5GB for Qwen2.5-1.5B)
**재방문**: IndexedDB 캐시에서 로드 → 훨씬 빠름

## 장점

| 항목 | 설명 |
|------|------|
| **서버 비용 제로** | GPU 서버 없이 사용자의 브라우저에서 실행. API 호출 비용이 없다 |
| **프라이버시** | 사용자 데이터가 서버로 전송되지 않음. 모든 처리가 로컬에서 완료 |
| **오프라인 사용** | 모델이 한 번 캐시되면 인터넷 없이도 사용 가능 |
| **OpenAI API 호환** | `chat.completions.create()` — 기존 OpenAI SDK 코드를 거의 그대로 사용 가능 |
| **스트리밍 지원** | `stream: true`로 토큰 단위 실시간 출력 |
| **Web Worker 지원** | 추론을 별도 스레드에서 실행 → UI 멈춤 없음 |
| **다양한 모델** | Llama, Phi, Gemma, Qwen, Mistral 등 주요 오픈소스 모델 지원 |
| **양자화 모델** | 4비트 양자화로 메모리 사용량 대폭 감소 (8B 모델도 ~4GB로 실행) |
| **설치 간단** | `npm install @mlc-ai/web-llm` 한 줄 + 코드 몇 줄로 동작 |
| **CDN 사용 가능** | npm 없이 `import("https://esm.run/@mlc-ai/web-llm")`로 즉시 사용 |
| **WebGPU 활용** | GPU 하드웨어 가속으로 CPU 대비 수십 배 빠른 추론 |

## 단점

| 항목 | 설명 | 심각도 |
|------|------|--------|
| **초기 모델 다운로드** | 1.5B 모델도 ~1.5GB 다운로드 필요. 사용자가 기다려야 함 | **높음** |
| **VRAM 제한** | 브라우저가 사용할 수 있는 GPU 메모리에 한계. 8B 이상 모델은 VRAM 부족 가능 | **높음** |
| **모델 품질 한계** | 브라우저에서 실행 가능한 모델은 1.5B~8B 수준. GPT-4/Claude 급 품질은 불가능 | **높음** |
| **WebGPU 브라우저 제한** | Chrome 113+, Edge 113+ 필수. Safari/Firefox는 실험적이거나 미지원 | **중간** |
| **추론 속도** | 서버 GPU(A100/H100) 대비 느림. 소비자 GPU에서 ~10-30 tok/s 수준 | **중간** |
| **첫 토큰 지연 (TTFT)** | 모델 로드 후 첫 추론 시 셰이더 컴파일로 몇 초 지연 가능 | **중간** |
| **메모리 소비** | 브라우저 탭이 수 GB RAM/VRAM 사용. 다른 탭이나 3D 앱과 동시 사용 시 문제 | **중간** |
| **모바일 미지원** | 모바일 브라우저의 WebGPU 지원이 제한적. 실질적으로 데스크톱 전용 | **중간** |
| **디버깅 어려움** | WASM + WebGPU 레이어 깊숙이 있어 에러 발생 시 원인 파악 어려움 | **낮음** |
| **모델 커스터마이징** | 새 모델 추가 시 MLC 컴파일 파이프라인을 거쳐야 함 (간단하지 않음) | **낮음** |

## Claude/GPT와 뭐가 다른가?

### 한 줄 요약

> **Claude/GPT = 식당에서 셰프가 요리해서 가져다주는 것**
> **WebLLM = 식재료를 집에 배달받아 내가 직접 요리하는 것**

### 구조 비교

```
┌─ Claude / GPT (서버 기반) ───────────────────────────────────────┐
│                                                                   │
│  사용자 브라우저 ──── 인터넷 ────► Anthropic/OpenAI 서버          │
│  (입력 전송)          ↑              │                            │
│                       │              ▼                            │
│                       │         GPU 클러스터 (A100/H100 수천 장)  │
│                       │         모델: 수백B ~ 수조 파라미터       │
│                       │              │                            │
│                       └──────────────┘                            │
│                        (응답 반환)                                │
│                                                                   │
│  모델 크기: 비공개 (추정 수백B~1T+)                               │
│  하드웨어: 데이터센터 GPU 클러스터                                │
│  비용: API 호출당 과금 ($0.01~$0.10/요청)                        │
│  데이터: 서버로 전송됨                                            │
└───────────────────────────────────────────────────────────────────┘

┌─ WebLLM (브라우저 로컬) ─────────────────────────────────────────┐
│                                                                   │
│  사용자 브라우저 (모든 것이 여기서 실행)                          │
│  ├─ 모델 파일 (IndexedDB에 캐시)                                 │
│  ├─ WebGPU → 사용자 PC의 GPU 1장                                │
│  ├─ WASM 런타임                                                  │
│  └─ 추론 결과 → 바로 화면에 표시                                 │
│                                                                   │
│  서버 통신: 없음 (오프라인 가능)                                  │
│  모델 크기: 0.5B ~ 8B (양자화)                                   │
│  하드웨어: 사용자 PC GPU 1장                                     │
│  비용: 무료 (전기세만)                                           │
│  데이터: 브라우저 밖으로 나가지 않음                              │
└───────────────────────────────────────────────────────────────────┘
```

### 상세 비교

| 항목 | Claude / GPT | WebLLM |
|------|-------------|--------|
| **실행 위치** | 클라우드 GPU 서버 | 사용자 브라우저 (로컬 GPU) |
| **모델 크기** | 수백B~1T+ 파라미터 | 0.5B~8B 파라미터 |
| **응답 품질** | 최상급 — 복잡한 추론, 코딩, 창작 가능 | 기본적 — 간단한 대화, 요약, 번역 수준 |
| **한국어 능력** | 매우 우수 | 모델에 따라 제한적 (영어 중심 학습) |
| **추론 속도** | 매우 빠름 (~100+ tok/s) | 느림 (~10-30 tok/s) |
| **비용** | API 호출당 과금 | 무료 (전기세만) |
| **프라이버시** | 데이터가 서버로 전송됨 | 데이터가 브라우저 밖으로 나가지 않음 |
| **오프라인** | 불가능 (인터넷 필수) | 가능 (모델 캐시 후) |
| **초기 설정** | 즉시 사용 가능 | 모델 다운로드 필요 (~1.5GB+) |
| **컨텍스트 길이** | 100K~200K 토큰 | ~4K 토큰 (메모리 한계) |
| **멀티모달** | 이미지/코드/파일 분석 가능 | 텍스트만 |
| **도구 사용** | Function calling, 웹 검색, 코드 실행 | Function calling (제한적) |
| **동시 사용자** | 수백만 | 1명 (브라우저 탭 1개) |
| **가용성** | 서버 장애/점검 시 사용 불가 | 항상 사용 가능 (오프라인 포함) |

### 품질 차이가 나는 근본적 이유

```
모델 크기 차이:
  Claude/GPT   ████████████████████████████████████████  수백B+ 파라미터
  Llama-8B     ████                                      8B 파라미터
  Qwen-1.5B    █                                         1.5B 파라미터

파라미터가 많을수록:
  → 더 많은 지식을 저장
  → 더 복잡한 패턴을 학습
  → 더 긴 맥락을 이해
  → 더 정교한 추론 가능

하드웨어 차이:
  서버: A100 80GB × 8장 = 640GB VRAM → 수백B 모델 실행 가능
  브라우저: RTX 4060 8GB × 1장 = 8GB VRAM → 8B 모델이 한계
```

> **결론**: WebLLM은 Claude/GPT를 "대체"하는 게 아니다.
> **용도가 다르다** — 프라이버시가 중요하거나, 오프라인이 필요하거나, 서버 비용을 쓸 수 없는 경우에 쓴다.
> 복잡한 추론/코딩/분석이 필요하면 여전히 Claude/GPT가 압도적이다.

## WebGPU + 브라우저 AI — 이게 의미하는 방향

### 왜 지금 이게 가능해졌나

WebLLM 같은 "브라우저에서 AI 모델 실행"이 2024~2025년에 갑자기 가능해진 이유는 **3가지 기술이 동시에 성숙**했기 때문이다.

```
2023~2025년에 동시에 일어난 일:

1. WebGPU 표준화 (2023.04 Chrome 113)
   └─ 브라우저에서 GPU Compute Shader 사용 가능
   └─ WebGL로는 불가능했던 "범용 GPU 연산"이 가능해짐

2. 소형 LLM의 품질 향상
   └─ Llama 2 → 3, Phi-2 → 3.5, Qwen 1.5 → 2.5, Gemma → 2
   └─ 1B~8B 모델도 "쓸 만한" 수준에 도달

3. 양자화 기술 발전
   └─ 모델을 4비트로 압축해도 품질 저하 최소화
   └─ 8B 모델: 32GB → 4GB로 축소 → 소비자 GPU에서 실행 가능
```

| 기술 | 없었을 때 | 생긴 후 |
|------|----------|---------|
| **WebGPU** | 브라우저에서 GPU 연산 불가 (WebGL = 그래픽 전용) | Compute Shader로 행렬곱 가능 → LLM 추론 가능 |
| **소형 LLM** | GPT-3(175B)급만 쓸 만함 → 서버 필수 | 1.5B~8B도 간단한 작업은 충분 |
| **양자화** | 8B 모델 = 32GB VRAM 필요 → 서버 GPU 필수 | 8B 모델 = 4GB → 노트북 GPU에서 실행 가능 |

> **WebGPU가 핵심 열쇠**: WebGL 시대에는 "3D 그래픽"만 가능했지만,
> WebGPU의 **Compute Shader**가 "범용 GPU 연산"을 열면서 브라우저가 AI 추론 플랫폼이 됐다.
> 우리가 Orillusion에서 테스트한 것과 동일한 WebGPU 기술이 AI 추론에도 쓰이는 것이다.

### 추후 방향 예측

#### 단기 (1~2년)

| 방향 | 설명 |
|------|------|
| **Chrome 내장 AI (Built-in AI)** | Google이 이미 Chrome에 Gemini Nano를 내장 중. 별도 다운로드 없이 브라우저 자체에 모델이 포함됨 |
| **모델 품질 향상** | 2B~4B 모델의 품질이 현재 8B 수준으로 향상. 브라우저에서 "쓸 만한" 범위가 넓어짐 |
| **WebGPU 브라우저 확대** | Safari, Firefox도 WebGPU 정식 지원. 크로스 브라우저 AI가 가능해짐 |
| **하이브리드 모드** | 간단한 작업은 로컬, 복잡한 작업은 서버 — 자동 분기 아키텍처 |

#### 중기 (3~5년)

| 방향 | 설명 |
|------|------|
| **엣지 AI 웹앱** | 서버 없이 브라우저만으로 AI 기능이 포함된 웹앱 배포. 스타트업 진입 장벽 대폭 하락 |
| **프라이버시 퍼스트 AI** | 의료/법률/금융 등 민감 데이터를 서버로 보내지 않는 AI 서비스 |
| **P2P AI 네트워크** | 여러 브라우저의 GPU를 연결해서 더 큰 모델을 분산 추론 (실험 단계) |
| **AI + 3D 융합** | WebGPU로 3D 렌더링 + AI 추론을 동시에 — 지능형 3D 캐릭터, 실시간 NPC 대화 등 |

#### 장기 (5년+)

| 방향 | 설명 |
|------|------|
| **브라우저 = AI 런타임** | 지금 브라우저가 "앱 실행 플랫폼"인 것처럼, "AI 실행 플랫폼"으로 진화 |
| **서버리스 AI** | 소규모 AI 기능은 서버 없이 프론트엔드만으로 완결. 백엔드는 복잡한 작업만 담당 |

### 큰 그림: 컴퓨팅의 패턴이 반복된다

```
1990년대: 연산은 서버에서 → 클라이언트는 터미널
2000년대: 웹 브라우저가 발전 → 클라이언트에서 점점 더 많은 연산
2010년대: SPA/PWA → "앱 수준"의 복잡한 로직이 브라우저에서 실행
2020년대: WebGPU + AI → "AI 추론"까지 브라우저에서 실행 ← 지금 여기

패턴: "서버에서만 가능하던 것"이 시간이 지나면 "클라이언트에서도 가능"해진다.
AI도 같은 패턴을 따르고 있다.
```

> WebLLM은 이 패턴의 **초기 단계**다.
> 지금은 "간단한 작업만 로컬에서" 수준이지만,
> GPU 성능 향상 + 모델 경량화 + WebGPU 성숙으로 점점 더 많은 AI 작업이 브라우저로 내려올 것이다.

### 어디에 활용할 수 있는가

#### 지금 당장 쓸 수 있는 곳

| 활용 | 설명 | 왜 WebLLM이 적합한가 |
|------|------|---------------------|
| **오프라인 문서 도우미** | 비행기/지하철 등 오프라인 환경에서 문서 요약, 번역 | 인터넷 불필요 |
| **프라이버시 민감 입력** | 의료 기록, 법률 문서, 개인 일기 등의 분석 | 데이터가 서버로 안 나감 |
| **폼 입력 보조** | 웹사이트의 텍스트 입력란에서 자동완성, 문법 교정 | 실시간 + 서버 비용 없음 |
| **교육용 AI 챗봇** | 학습 도우미, 퀴즈 생성, 설명 생성 | 학교/기관의 서버 비용 부담 없음 |
| **Chrome 확장 프로그램** | 웹페이지 요약, 이메일 초안 작성, 번역 | 확장 프로그램에 AI를 내장 |
| **프로토타이핑** | AI 기능 PoC를 서버 인프라 없이 빠르게 구현 | GPU 서버 세팅 없이 즉시 시작 |

#### 이 프로젝트(webgpuTest)와의 연결

| 시나리오 | 설명 |
|---------|------|
| **3D + AI 결합** | Orillusion 3D 씬 안의 NPC가 WebLLM으로 대화하는 데모 — 서버 없이 지능형 3D 캐릭터 |
| **3D 모델 설명 생성** | 사용자가 보고 있는 3D 모델에 대한 설명을 로컬 AI가 생성 |
| **음성 → 3D 제어** | STT(음성인식) + WebLLM(의도 파악) → 3D 씬 제어 ("차를 빨간색으로 바꿔줘") |

> **주의**: 3D 렌더링과 LLM 추론이 **동일 GPU를 공유**하므로, 동시 실행 시 VRAM 경합이 발생한다.
> 실험적 용도로는 가능하지만, 프로덕션에서는 리소스 분배를 신중히 고려해야 한다.

#### 쓰지 말아야 하는 곳

| 상황 | 이유 |
|------|------|
| **복잡한 추론/코딩이 필요한 경우** | 1.5B~8B 모델로는 품질 부족 → Claude/GPT 사용 |
| **대량 처리 (배치)** | 브라우저 1개 = 사용자 1명. 서버에서 병렬 처리가 효율적 |
| **모바일 사용자 타겟** | 모바일 WebGPU 지원 미비 + VRAM 부족 |
| **엔터프라이즈 서비스** | 모델 품질/안정성/SLA 보장이 필요하면 클라우드 API가 적합 |
