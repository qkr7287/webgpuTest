# 3D 엔진 비교 종합 정리

> **테스트 기반**: Babylon.js (WebGL) vs Three.js (WebGL) vs Orillusion (WebGPU)
> **테스트 모델**: `free_cyberpunk_hovercar.glb` (~27MB, 126 메시)
> **환경**: LG Gram → LAN → Ubuntu Docker 서버 (192.168.0.16)
> **브라우저**: Chrome 시크릿 모드 + 강력 새로고침 (캐시 완전 제거)

## 1. Widget 분리가 확실히 효과가 있는가?

### 결론: "극적인 차이는 없다. 하지만 분리가 손해를 보지도 않는다."

| 엔진 | web-host (분리) | combined (통합) | 차이 | 더 빠른 쪽 |
|------|----------------:|----------------:|-----:|-----------|
| Babylon.js | 3,558ms | 3,616ms | 58ms | **분리** |
| Three.js | 2,682ms | 2,369ms | 313ms | **통합** |

#### 수치로 보면

- **Babylon.js**: 분리(3,558ms)가 통합(3,616ms)보다 **58ms 빠름**
  - 3회 측정 편차: 분리 3,385~3,738 / 통합 3,489~3,833
  - 58ms 차이는 측정 편차(353ms, 344ms) 안에 들어가므로 **통계적으로 유의미하지 않음**
  - 다만, 3회 모두 분리가 통합보다 빨랐음 (3,385<3,489 / 3,551>3,526 / 3,738<3,833) → 2승 1패

- **Three.js**: 통합(2,369ms)이 분리(2,682ms)보다 **313ms 빠름**
  - 3회 측정 편차: 분리 2,247~3,506 / 통합 2,101~2,662
  - **분리 1회차(3,506ms)가 이상치** — 2·3회차(2,293, 2,247)는 통합과 거의 동등
  - 1회차 제외 시 분리 평균 = 2,270ms → 통합(2,369ms)과 **거의 동일하거나 오히려 빠름**

### 왜 차이가 작은가?

**병목이 27MB GLB 전송 (전체의 40~60%)**이기 때문이다.

```
embed.js fetch 오버헤드 = ~50-150ms = 전체의 3~5%
27MB GLB 전송           = ~500-2500ms = 전체의 40~60%
```

embed.js를 분리하든 통합하든, 진짜 병목인 GLB 전송 시간은 동일하다.

### Three.js 분리 1회차(3,506ms)는 왜 튀었나?

Three.js web-host 1회차만 3,506ms로 나머지(2,293, 2,247)와 **1,200ms 이상 차이**가 난다.

가능한 원인:
1. **Wi-Fi 순간 대역폭 저하** — 27MB 전송 시간이 ~1초 이상 증가
2. **Docker 콜드 스타트** — 첫 요청 시 Node.js/Vite 프로세스 웜업
3. **브라우저 첫 시크릿 윈도우** — V8 엔진 JIT 컴파일러 웜업

이 이상치를 제외하면 Three.js도 **분리 ≈ 통합**이다.

### 분리가 유리한 경우

| 조건 | 이유 |
|------|------|
| **번들이 큰 엔진** (Babylon.js) | 분리 시 호스트 JS가 작아 빠르게 파싱 → embed.js 병렬 fetch 가능 |
| **CDN 배포** | widget을 엣지 캐시에 올리면 재방문/다른 서비스 재사용 시 캐시 히트 |
| **여러 서비스에서 동일 위젯 재사용** | 같은 URL이면 브라우저 캐시 공유 |
| **독립 배포 필요** | 위젯만 업데이트 가능 (호스트 재배포 불필요) |

### 통합이 유리한 경우

| 조건 | 이유 |
|------|------|
| **번들이 작은 엔진** (Three.js) | 통합 번들의 불이익이 적고, embed.js fetch RTT 제거 효과가 더 큼 |
| **운영 단순화** | CORS/CSP 설정 불필요, 서버 1대로 충분 |
| **단일 서비스** | 재사용 이점이 없으므로 복잡도만 늘어남 |

### 핵심

> **Widget 분리의 가치는 "로딩 속도"가 아니라 "아키텍처"에 있다.**
> - 독립 배포, 재사용, CDN 캐시가 필요하면 **분리**
> - 단일 서비스이고 운영 단순화가 중요하면 **통합**
> - 로딩 속도 차이는 어느 쪽이든 **전체의 3~5% 수준**으로 무의미

## 2. 어느 엔진이 더 빠른가? 왜?

### 순위 (초기 로딩 시간 기준, 평균)

```
1위  Three.js combined     2,369ms  ████████████
2위  Three.js web-host     2,682ms  █████████████▌
3위  Babylon.js web-host   3,558ms  █████████████████▊
4위  Babylon.js combined   3,616ms  ██████████████████
5위  Orillusion (WebGPU)   7,183ms  ████████████████████████████████████
```

### 엔진 단위 비교 (최선 모드 기준)

| 엔진 | 최선 모드 | 평균 | 최소값 | 최대값 | Babylon 대비 |
|------|----------|-----:|-------:|-------:|:------------|
| **Three.js** | combined | **2,369ms** | 2,101 | 2,662 | **1.53배 빠름** |
| **Babylon.js** | web-host | **3,558ms** | 3,385 | 3,738 | 기준 |
| **Orillusion** | 단일 앱 | **7,183ms** | 6,828 | 7,673 | **2.02배 느림** |

### Three.js가 가장 빠른 이유

| 요인 | Three.js | Babylon.js |
|------|----------|-----------|
| **코어 번들 크기** | 작음 (~600KB gzip) | 큼 (~1MB+ gzip) |
| **초기화 방식** | 가볍고 즉시적 | 더 많은 내부 시스템 초기화 |
| **GLB 파서** | 경량 (GLTFLoader) | 풍부하지만 무거움 (SceneLoader) |
| **Tree-shaking** | 효과적 (ESM 설계) | 상대적으로 제한적 |

Three.js는 "필요한 것만 가져오는" 철학으로, 단일 모델 로딩 시나리오에서 오버헤드가 가장 적다.

### Orillusion이 느린 이유 (3가지 겹침)

**1) WebGPU 초기화 오버헤드 (본질적)**

| 단계 | WebGL | WebGPU |
|------|-------|--------|
| 컨텍스트 | `getContext('webgl')` 동기 | `requestAdapter()` + `requestDevice()` 비동기 |
| 셰이더 | 첫 drawcall 시 암묵적 | 파이프라인 생성 시 **사전 컴파일** (WGSL→바이너리) |
| 파이프라인 | GL state machine | `createRenderPipeline()` 명시적 |

WebGPU는 **런타임 jank 제거를 위해 초기화 비용을 앞당긴** 설계다. 대신 렌더링 중 프레임 드롭이 적다.

**2) 엔진 미성숙 (v0.8.5-dev.9 알파)**

- GLB 파서 버그로 **수동 패치 필요** (`glbNodeAnimation.ts`에서 직접 GLB 바이너리 파싱)
- 비골격 노드 애니메이션 미지원 → 커스텀 런타임으로 보완
- `GLTF_NODE_INDEX_PROPERTY`가 `undefined`로 선언되어 `obj['undefined']`에 nodeId 저장하는 버그

**3) 테스트 조건 불이익**

- **비압축 소스** 사용 (~2.16MB `orillusion.es.max.js`) — Vite prebundling 버그 우회
- **추가 작업**: HDR 큐브맵 + SkyRenderer + Bloom + TAA + 커스텀 노드 애니메이션 파서 (GLB 이중 fetch)

### Orillusion 7,183ms 분해 추정

```
┌───────────────────────────────────────────────────────────┐
│  2.16MB JS 전송+파싱               ~300-500ms             │
│  WebGPU 초기화 (adapter+device)    ~200-400ms             │
│  HDR 큐브맵 로드+SkyRenderer       ~200-400ms  ← 추가 작업│
│  GLB 27MB 전송 (LAN)              ~500-2500ms ← 최대 병목 │
│  GLB 파싱 + GPU 파이프라인 생성    ~500-1000ms             │
│  Bloom+TAA 포스트프로세싱 설정     ~100-200ms  ← 추가 작업│
│  커스텀 노드 애니메이션 파서       ~200-500ms  ← GLB 이중 fetch│
│  WGSL 셰이더 사전 컴파일          ~300-600ms             │
│  30프레임 FPS 안정화 대기          ~500-1000ms             │
│                                                           │
│  총합: ~2,800ms ~ 7,100ms                                 │
│  실측 평균: 7,183ms                                        │
└───────────────────────────────────────────────────────────┘
```

### 하지만 WebGPU가 빛나는 영역은 다르다

| 시나리오 | WebGL | WebGPU |
|---------|-------|--------|
| 단일 모델 로딩 속도 | **빠름** | 느림 |
| 100+ 오브젝트 렌더링 FPS | CPU 바운드 시작 | 여유 있음 |
| 1000+ 드로우콜 | **CPU 병목 심각** | CPU 오버헤드 대폭 감소 |
| GPU Compute | 불가 | **네이티브 지원** |
| 파티클 100만 개 | CPU 한계 | Compute Shader로 처리 |

> 이번 테스트는 "GLB 1개 로드 → 안정화 시간"을 측정했으므로 **WebGPU의 장점이 전혀 드러나지 않는 영역**이다.

## 3. 로딩 속도 편차가 클 때는 무엇 때문인가?

### 현상 — 실측 데이터에서 확인되는 편차

| 엔드포인트 | 최소 | 최대 | **편차 (max-min)** | 편차율 |
|-----------|-----:|-----:|-------------------:|-------:|
| 5100 (webHost_babylon) | 3,385 | 3,738 | 353ms | 10% |
| 5102 (combine_babylon) | 3,489 | 3,833 | 344ms | 10% |
| 5200 (webHost_three) | 2,247 | **3,506** | **1,259ms** | **47%** |
| 5202 (combine_three) | 2,101 | 2,662 | 561ms | 24% |
| 5300 (orillusion) | 6,828 | 7,673 | 845ms | 12% |

**Three.js web-host의 1회차(3,506ms)**가 눈에 띄게 튀었고, 동일 조건에서도 **3초 vs 8초** 수준의 편차가 관찰된 적이 있다.

### 원인 순위

#### 1순위: Wi-Fi 대역폭 변동 (가장 유력)

매번 27MB를 네트워크로 풀 다운로드하므로 Wi-Fi 상태가 로딩 시간을 **직접 결정**한다.

```
평소:  Wi-Fi 300Mbps → GLB ~0.7초 → 총 ~3초
가끔:  Wi-Fi  50Mbps → GLB ~4.3초 → 총 ~8초
                        ↑
                    이 차이만으로 5초 벌어짐
```

Wi-Fi 간섭 원인:
- 같은 채널의 옆집 공유기
- 같은 네트워크의 다른 기기 대용량 다운로드
- 블루투스 간섭 (2.4GHz)
- 사람 이동에 의한 순간 신호 감쇠
- 전자레인지 등 전자파

#### 2순위: LG Gram CPU 서멀 스로틀링

```
정상: CPU 4.0GHz → JS 파싱 + GLB 디코딩 빠름 → 3초
과열: CPU 1.8GHz → 처리 속도 절반             → 6~8초
```

얇은 노트북이라 직전 작업에 따라 CPU 클럭이 크게 변동.

#### 3순위: Docker 컨테이너 리소스 경합

4개 컨테이너가 같은 CPU/메모리/디스크 I/O를 공유.
다른 컨테이너가 동시에 요청을 처리하거나 Node.js GC가 돌면 디스크 I/O / CPU 경합 발생.

#### 4순위: V8 GC (상대적으로 영향 적음)

27MB ArrayBuffer 파싱 시 수백MB 임시 객체 → Major GC 시 수백ms 멈춤 가능.
다만 시크릿 모드는 매번 깨끗한 힙에서 시작하므로 영향은 상대적으로 적다.

### 빠른 진단법

Network 탭에서 `free_cyberpunk_hovercar.glb`의 Time만 비교:
- **GLB Time이 길면** → 네트워크 문제
- **GLB Time은 비슷한데 전체가 느리면** → CPU/GC 문제

### 편차 줄이는 법

| 방법 | 효과 | 난이도 |
|------|------|-------|
| **유선 LAN으로 테스트** | Wi-Fi 변동 완전 제거 (가장 확실) | 낮음 |
| GLB gzip/brotli 압축 서빙 | 27MB → ~5-8MB 전송량, 네트워크 영향 감소 | 중간 |
| 테스트 전 노트북 5분 쿨다운 | CPU 서멀 스로틀링 방지 | 낮음 |
| 불필요한 Docker 컨테이너 중지 | 서버 리소스 경합 감소 | 낮음 |
| 5회 이상 측정, 최고/최저 제외 평균 | 통계적 신뢰도 향상 | 낮음 |

## 4. 프로젝트 성향별 엔진 선택 가이드

### 한눈에 보는 선택 기준

| 프로젝트 성향 | 추천 엔진 | 이유 |
|-------------|----------|------|
| **제품 뷰어 / 쇼케이스** (모델 1~5개) | **Three.js** | 가장 빠른 초기 로딩 (2,369ms), 가벼운 번들, 풍부한 생태계 |
| **인터랙티브 웹앱** (중간 복잡도) | **Three.js** | React Three Fiber 등 프레임워크 통합 우수 |
| **대규모 씬** (100+ 오브젝트, 복잡한 포스트프로세싱) | **Babylon.js** | 빌트인 물리엔진, GUI, Inspector, 프로덕션 검증 |
| **게임 / 시뮬레이션** (물리, 충돌, AI) | **Babylon.js** | 게임 엔진 수준의 통합 기능 |
| **대량 렌더링** (1000+ 드로우콜, 파티클 시뮬) | **Orillusion (WebGPU)** | GPU Compute, CPU 오버헤드 최소, 미래 지향 |
| **GPU Compute 활용** (ML 추론, 물리 시뮬) | **Orillusion (WebGPU)** | Compute Shader 네이티브 |
| **위젯 / 임베드** (외부 사이트에 삽입) | **Three.js** | 번들 크기 최소, tree-shaking 효과적 |
| **빠른 프로토타이핑** | **Three.js** | 낮은 러닝커브, 방대한 예제/커뮤니티 |
| **엔터프라이즈 / 장기 프로젝트** | **Babylon.js** | Microsoft 지원, 안정적 LTS, 문서 품질 |

### 엔진별 상세 비교

| 항목 | Three.js | Babylon.js | Orillusion |
|------|----------|-----------|-----------|
| **렌더링 API** | WebGL (2.0) | WebGL (2.0) | WebGPU |

## 5. WebGPU 테스트 — Orillusion 사용기

### 장점

| 항목 | 설명 |
|------|------|
| **WebGPU 네이티브** | WebGL 래핑이 아닌 처음부터 WebGPU 전용 설계. GPU 파이프라인/셰이더를 직접 제어 |
| **Compute Shader 지원** | 파티클, 물리 시뮬레이션 등을 GPU에서 직접 처리 가능 |
| **포스트 프로세싱 내장** | Bloom, TAA 등을 엔진 레벨에서 바로 적용 가능. 설정도 비교적 간단 |
| **컴포넌트 기반 아키텍처** | Unity 스타일의 `addComponent()` 패턴. 익숙하면 직관적 |
| **HDR 환경맵 로드** | `loadHDRTextureCube()` 한 줄로 HDR 큐브맵 적용. SkyRenderer도 간단 |
| **TypeScript 퍼스트** | 소스가 TS로 작성되어 타입 지원이 기본 |
| **렌더링 중 프레임 안정성** | WebGPU 특성상 셰이더 사전 컴파일로 렌더링 중 jank가 적음 |

### 단점

| 항목 | 설명 | 심각도 |
|------|------|--------|
| **GLB 파서 버그** | 비골격(노드) 애니메이션을 처리하지 못함. `GLTF_NODE_INDEX_PROPERTY`가 `undefined`로 선언되어 `obj['undefined']`에 nodeId를 저장하는 버그. 결국 GLB를 직접 바이너리 파싱하는 커스텀 런타임(`glbNodeAnimation.ts`)을 작성해야 했음 | **높음** |
| **Vite prebundling 충돌** | npm 패키지를 정상적으로 import하면 Vite가 prebundle 시 에러 발생. 비압축 소스(`orillusion.es.max.js` 2.16MB)를 `src/lib/`에 수동 복사해서 우회해야 했음 | **높음** |
| **문서/예제 부족** | 공식 문서가 빈약하고, 에러 발생 시 소스 코드를 직접 읽어야 함. 커뮤니티가 작아서 Stack Overflow 등에서 답을 찾기 어려움 | **높음** |
| **알파 버전 (v0.8.5-dev.9)** | API가 안정되지 않음. 메이저 업데이트 시 breaking change 가능성 높음 | **중간** |
| **초기 로딩 느림** | WebGPU 특성 + 엔진 미최적화로 초기 로딩이 WebGL 대비 2~3배 느림 (7,183ms vs 2,369ms) | **중간** |
| **브라우저 호환성** | WebGPU를 지원하는 브라우저에서만 실행 가능 (Chrome 113+, Edge 113+, Firefox는 실험적) | **중간** |
| **TypeScript 타입 불일치** | 엔진 내부 타입과 실제 동작이 다른 경우가 있어 `as any` 캐스팅이 빈번하게 필요 | **낮음** |

### 실제 겪은 삽질 요약

1. **GLB 애니메이션 재생 실패** → Orillusion의 GLB 파서가 노드 애니메이션을 무시 → 직접 GLB 바이너리를 파싱해서 `translation/rotation/scale` 채널을 추출, 매 프레임 transform 갱신하는 커스텀 컴포넌트 작성
2. **노드 탐색 실패** → `GLTF_NODE_INDEX_PROPERTY`가 `undefined`라서 모든 노드가 `obj['undefined']` 키에 저장됨 → 노드 이름 기반 탐색으로 전환
3. **회전 적용 안됨** → `localRotQuat`에 인라인 `set()`을 써도 dirty 플래그가 안 세워짐 → setter 할당 방식으로 변경
4. **Vite에서 import 실패** → npm 패키지 그대로 쓰면 prebundling 에러 → 소스 파일을 프로젝트 내부에 복사
5. **TAAPost/BloomPost 타입 에러** → `addPost()`/`removePost()`가 인스턴스가 아닌 클래스를 받는데 TS 타입이 안 맞음 → `as any` 처리

### Orillusion을 써야 하는가?

| 상황 | 판단 |
|------|------|
| WebGPU의 Compute Shader가 반드시 필요한 R&D/프로토타입 | **써볼 만하다** |
| 대량 렌더링 성능 실험 | **써볼 만하다** |
| 프로덕션 서비스 | **아직 이르다** — GLB 파서 버그, API 불안정, 문서 부족 |
| WebGPU 학습 목적 | **괜찮다** — WebGPU 파이프라인 구조를 이해하기에는 나쁘지 않음 |
