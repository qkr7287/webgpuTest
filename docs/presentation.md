## 발표 개요

Web 기반 3D 기술의 현재와 미래를 탐구하기 위해 4가지 주제로 실제 프로젝트를 구성하고 테스트를 진행하였습니다.

| 주제 | 핵심 질문 |
|------|-----------|
| 1. 개발 구조 분리 | Web+3D 합쳐진 프로젝트를 분리할 수 있는가? |

| 3. Orillusion (WebGPU) | 차세대 엔진으로 자리잡을 수 있는가? |
| 4. WebLLM | 브라우저에서 LLM을 활용할 수 있는가? |

## 1. 개발 구조 분리 — Web+3D 프로젝트 분리 테스트

### 문제 인식

Web + 3D가 합쳐진 프로젝트는 개발/배포/팀 협업 측면에서 불편합니다. 3D 위젯과 Web 프로젝트를 독립적으로 분리할 수 없는가?

### 테스트 방법

**3d-widget-web-host** 프로젝트를 구성하여 테스트하였습니다.

- **기술**: Remote ESM Module Import (iframe이 아닌 동적 모듈 임포트)
- **구조**: 3D Widget 서버와 Web Host 서버를 분리하고, Host에서 Widget의 mount 함수를 원격으로 import
- **비교 대상**: 분리 구조(web-host) vs 통합 구조(combined-app)
- **테스트 모델**: `free_cyberpunk_hovercar.glb` (~27MB, 126 meshes)
- **환경**: LAN (192.168.0.x, Ubuntu Docker), Chrome 시크릿 모드, 캐시 초기화

### 로딩 타임 측정 결과

| 엔진 | 분리 구조 (web-host) | 통합 구조 (combined) | 차이 |
|------|---------------------|---------------------|------|
| Three.js | **2,401ms** | 2,651ms | -250ms (분리가 빠름) |
| Babylon.js | **3,558ms** | 3,616ms | -58ms (분리가 빠름) |

### 결론

- **양쪽 엔진 모두 분리 구조가 더 빠른 결과**
- 로딩타임 최적화 관점에서 극적인 차이는 아니지만, **열악한 환경의 PC에서는 그 차이가 많이 벌어지는 것으로 확인**
- 번들 분할에 의한 병렬 로딩 효과가 embed.js fetch 오버헤드를 상쇄
- **유의미한 결과**: 성능 손실 없이 개발 구조 분리가 가능하며, 팀 단위 개발/배포 독립성을 확보할 수 있음

### 분리 구조의 이점

- Web 팀과 3D 팀의 독립적 개발/배포 가능
- 3D Widget을 CDN에 배포하여 여러 서비스에서 재사용 가능
- 버전 관리 및 롤백이 개별적으로 가능

## 2. Babylon.js vs Three.js — 엔진 비교 분석

### 테스트 방법

동일한 GLB 모델(`free_cyberpunk_hovercar.glb`)을 양쪽 엔진에 기본 설정으로 로드하여 **퀄리티**와 **로딩 타임**을 비교하였습니다.

### 로딩 타임 비교

| 엔진 | 평균 로딩 시간 | 번들 크기 |
|------|--------------|----------|
| Three.js | **2,401ms** | ~600KB (gzip) |
| Babylon.js | **3,558ms** | ~1MB (gzip) |

Three.js가 약 **1.48배 빠름**.

### 퀄리티 비교

**기본 설정 기준으로 Three.js가 더 좋은 퀄리티**를 보여줌.

이유: 두 엔진의 **내부 라이팅/렌더링 로직이 근본적으로 다름**

- **Three.js**: 기본 톤매핑과 라이팅 파이프라인이 바로 사용하기에 최적화
- **Babylon.js**: PBR 파이프라인이 더 정교하지만, 기본 설정만으로는 그 장점이 드러나지 않음

### 프레임 안정성 (동일 인스턴스 대량 소환 테스트)

**같은 인스턴스를 다수 소환하여 프레임 안정성을 비교한 결과, Babylon.js가 우위**

- Babylon.js는 많은 객체를 다룰 때 프레임 드롭이 적고 안정적
- 고성능 퀄리티의 복잡한 씬에서 Babylon.js의 최적화가 빛을 발함

### 왜 차이가 나는가?

같은 "3D 엔진"이라고 같은 것이 아닙니다. **비슷할 뿐이지 내부 로직에 따른 차이가 엄청납니다.**

| 항목 | Three.js | Babylon.js |
|------|----------|------------|
| 라이팅 시각화 | 기본 설정이 직관적, 바로 좋은 퀄리티 | 세밀한 설정이 필요, 설정 시 더 높은 퀄리티 가능 |
| 같은 모델 로드 | 기본 구성으로 더 나은 결과 | 추가 설정 후 동등 이상 |
| 대량 인스턴스 | 프레임 드롭 발생 | 프레임 안정성 우수 |
| 번들 크기 | 가벼움 (~600KB) | 무거움 (~1MB) |

### 결론 — 적재적소

| 상황 | 추천 엔진 |
|------|----------|
| 빠른 프로토타이핑, 간단한 3D 뷰어 | **Three.js** |
| 제품 쇼케이스, 소규모 씬 | **Three.js** |
| 대규모 씬, 많은 객체 | **Babylon.js** |
| 고성능 그래픽, 복잡한 렌더링 | **Babylon.js** |
| 게임 등 인터랙티브 콘텐츠 | **Babylon.js** |

이 테스트로 인해 **각 엔진의 장점이 명확해지고, 프로젝트 성격에 따라 적용할 엔진을 판단**할 수 있게 되었습니다.

## 3. Orillusion — WebGPU 차세대 엔진 테스트

### Orillusion이란?

- 중국에서 개발한 **WebGPU 전용 3D 엔진**
- 출시 약 2~3년 (현재 v0.8.5-dev.9, 알파 단계)
- WebGL이 아닌 **WebGPU만** 사용하는 차세대 엔진

### 로딩 타임 비교

| 엔진 | 평균 로딩 시간 | 배수 |
|------|--------------|------|
| Three.js | 2,401ms | 1x (기준) |
| Babylon.js | 3,558ms | 1.48x |
| **Orillusion** | **7,183ms** | **3x** |

### 왜 느린가?

1. **WebGPU 초기화 오버헤드**: WebGL의 동기식 컨텍스트 생성과 달리, WebGPU는 비동기 어댑터 요청 + 디바이스 요청 + 명시적 파이프라인 컴파일이 필요
2. **엔진 미성숙**: GLB 파서 버그 (수동 패치 필요), 일부 애니메이션 미지원
3. **최적화 부족**: 2.16MB 비압축 소스 (Vite 프리번들링 버그 우회)

### 그럼에도 주목할 점

퀄리티는 오랜 경력의 Three.js, Babylon.js가 현재로서는 더 나은 것으로 판단됩니다. 그러나:

- 현재 Babylon.js가 **프레임 안정성과 고성능 그래픽 엔진**으로 자리잡은 위치를 **위협할 만한 엔진**
- WebGPU의 **고성능 연산 및 병렬 처리 능력이 뛰어남**

| 시나리오 | WebGL (Three/Babylon) | WebGPU (Orillusion) |
|----------|----------------------|---------------------|
| 단일 모델 로드 | 빠름 | 느림 (초기화 오버헤드) |
| 100+ 객체 렌더링 | CPU 바운드 | 효율적 |
| 1000+ 드로우콜 | CPU 병목 | CPU 오버헤드 제거 |
| GPU 컴퓨트 셰이더 | 미지원 | 네이티브 지원 |
| 100만+ 파티클 | CPU 한계 | 컴퓨트 셰이더로 가능 |

### 결론

- **현재**: 프로덕션 투입에는 이르지만, R&D/실험 단계로 적합
- **미래**: WebGPU가 표준화되면 고성능 연산이 필요한 영역에서 강력한 대안이 될 가능성이 높음
- **주시 필요**: Babylon.js의 고성능 영역을 대체할 잠재력 보유

## 4. WebLLM — 브라우저에서 LLM 실행

### WebLLM이란?

- **웹 브라우저에서 WebGPU를 이용하여 LLM 모델을 로컬 실행**하는 기술
- 서버 없이 브라우저 GPU만으로 AI 추론 수행
- https://webllm.mlc.ai/ 에서 라이브러리 제공

### 동작 원리

```
[모델 제작사] Qwen, Llama 등 오픈소스 모델 학습
       ↓
[MLC-AI] 학습된 모델을 WebGPU용으로 변환 + JS 라이브러리 제공
       ↓
[브라우저] npm 패키지로 가져와서 WebGPU로 추론 실행
```

- 학습(Training)은 불가, **추론(Inference) 전용**
- GPT, Gemini, Claude 등 비공개 모델은 사용 불가 (가중치 비공개)
- Qwen, Llama 등 **오픈소스 모델만** 사용 가능

### 테스트 결과

- **모델 다운로드 속도**: 크게 느리지 않음
- **재접속 시**: 캐싱된 데이터로 초기 로드보다 빠름
- **추론 품질**: GPT, Gemini에 비할 바는 안 되지만, 웹에서 로컬로 LLM을 실행할 수 있다는 것에 의의

### 활용 예시 — 3D AI Scene

WebGPU + Three.js를 결합한 예시 프로젝트를 구현하였습니다.

```
사용자: "빨간 공을 만들어"
       ↓
[WebLLM] 시스템 프롬프트의 규칙에 따라 자연어 → JSON 변환
       {"action":"create","shape":"sphere","color":"#ff0000"}
       ↓
[Three.js] JSON 명령을 받아 3D 씬에 오브젝트 생성
```

- LLM은 "학습"이 아닌 **프롬프트 엔지니어링**으로 커맨드를 생성하는 번역기 역할
- 사용 가능한 액션 목록과 포맷을 시스템 프롬프트로 지정하면, LLM이 자연어를 정해진 JSON으로 변환

### 결론

- 웹에서 WebGPU를 이용해 LLM 모델을 실행할 수 있다는 것 자체가 의미 있는 기술
- 프라이버시가 중요한 환경(의료, 법률, 금융)에서 데이터가 서버로 나가지 않는 장점
- 오프라인 지원 (모델 캐싱 후)
- 3D + AI 결합 가능성 확인 (NPC 대화, 씬 제어 등)

## 종합 결론

| 주제 | 핵심 결과 |
|------|----------|
| 개발 구조 분리 | 양쪽 엔진 모두 분리가 빠름, 열악한 환경에서 차이 더 벌어짐. 팀 독립성 확보에 유의미 |
| Babylon vs Three | Three.js = 빠르고 기본 퀄리티 우수 / Babylon.js = 대규모 씬, 프레임 안정성 우수 |
| Orillusion | 현재는 미성숙하나, WebGPU 기반 고성능 연산에서 잠재력 확인 |
| WebLLM | 브라우저 로컬 LLM 실행 가능 확인. 3D+AI 결합 예시 구현 |

### 전체 시사점

1. **같은 3D 엔진이라고 같은 것이 아니다** — 내부 로직(라이팅, 렌더링, 최적화)에 따라 결과가 크게 다르며, 프로젝트 특성에 맞는 엔진 선택이 중요
2. **WebGPU는 미래 기술이지만, 지금도 주시할 가치가 있다** — 고성능 연산, 병렬 처리, LLM 추론 등 WebGL로 불가능한 영역을 열어줌
3. **아키텍처 분리는 성능이 아닌 구조적 이점이 핵심** — 네트워크 전송(GLB 27MB)이 전체 로딩의 40~60%를 차지하므로, 엔진이나 구조보다 에셋 최적화가 더 중요
