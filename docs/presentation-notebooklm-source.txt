Web 3D 엔진 및 WebGPU 기술 분석 보고서

이 문서는 Web 기반 3D 기술의 현재와 미래를 분석하기 위해 4가지 주제로 실제 프로젝트를 구성하고 테스트한 결과를 정리한 보고서입니다. 각 주제에 대해 실제 프로젝트를 만들어 테스트하고, 데이터를 측정하여 결론을 도출하였습니다.


제1장. 개발 구조 분리 — Web+3D 프로젝트 분리 테스트

문제 인식:
현재 Web과 3D가 하나의 프로젝트에 합쳐져 있는 구조는 개발, 배포, 팀 협업 측면에서 매우 불편합니다. Web 개발자와 3D 개발자가 동일 프로젝트에서 작업해야 하므로 충돌이 빈번하고, 배포도 통합 빌드가 필요합니다. 이 구조를 분리할 수 있는지, 분리하면 성능에 어떤 영향이 있는지를 검증하고자 했습니다.

테스트 환경:
3d-widget-web-host 프로젝트를 구성하였습니다. 이 프로젝트는 모노레포 구조로 Babylon.js와 Three.js 두 엔진에 대해 각각 3D Widget 서버, Web Host 서버, 통합 앱(combined-app) 3가지 형태를 구현하였습니다.

핵심 기술은 Remote ESM Module Import입니다. iframe이나 웹 컴포넌트가 아닌, 브라우저의 동적 ES Module import를 사용하여 원격 서버에서 3D 위젯의 mount 함수를 가져오는 방식입니다. 이렇게 하면 iframe의 격리 오버헤드 없이 동일 메모리 공간에서 3D를 실행할 수 있습니다.

테스트 모델은 free_cyberpunk_hovercar.glb로 약 27MB, 126개 메쉬로 구성된 GLB 파일입니다.
테스트 환경은 LAN 환경(192.168.0.x, Ubuntu Docker)에서 Chrome 시크릿 모드, 캐시 초기화 후 3회씩 측정하였습니다.

측정 결과:
Three.js 분리 구조: 평균 2,682ms (개별 측정 3,506ms, 2,293ms, 2,247ms)
Three.js 통합 구조: 평균 2,369ms (개별 측정 2,662ms, 2,345ms, 2,101ms)
Three.js는 통합이 313ms 빠름.

Babylon.js 분리 구조: 평균 3,558ms (개별 측정 3,385ms, 3,551ms, 3,738ms)
Babylon.js 통합 구조: 평균 3,616ms (개별 측정 3,489ms, 3,526ms, 3,833ms)
Babylon.js는 분리가 58ms 빠름.

분석:
로딩타임 최적화 관점에서 차이는 미미합니다. 전체 로딩의 40~60%가 27MB GLB 파일 네트워크 전송에 소요되므로, 분리 구조의 embed.js 추가 fetch는 3~5% 수준의 오버헤드에 불과합니다.

그러나 열악한 환경의 PC에서는 그 차이가 많이 벌어지는 것으로 확인되어, 유의미한 테스트였습니다. Babylon.js는 번들이 크기 때문에(약 1MB) 분리 시 병렬 로딩 이점이 생겨 오히려 빨라지는 결과가 나왔습니다.

결론:
성능 손실 없이 개발 구조 분리가 가능합니다. Web 팀과 3D 팀의 독립적 개발 및 배포가 가능해지고, 3D Widget을 CDN에 배포하여 여러 서비스에서 재사용할 수 있으며, 버전 관리와 롤백이 개별적으로 가능합니다. 실무적으로 충분히 적용 가능한 구조입니다.


제2장. Babylon.js vs Three.js 엔진 비교 분석

테스트 방법:
동일한 GLB 모델(free_cyberpunk_hovercar.glb)을 두 엔진의 기본 설정으로 로드하여 퀄리티와 로딩 타임을 비교하였습니다. 기본 설정이란 엔진을 초기화하고 모델을 로드하는 최소한의 코드만 사용한 것을 의미합니다.

로딩 타임 비교:
Three.js 평균 2,369ms, 번들 크기 약 600KB(gzip)
Babylon.js 평균 3,616ms, 번들 크기 약 1MB(gzip)
Three.js가 약 1.5배 빠릅니다.

퀄리티 비교:
기본 설정 기준으로 Three.js가 더 좋은 퀄리티를 보여주었습니다. 이는 두 엔진의 내부 라이팅 시각화 로직이 근본적으로 다르기 때문입니다.

Three.js는 기본 톤매핑과 라이팅 파이프라인이 바로 사용하기에 최적화되어 있어, 같은 모델을 로드해도 기본 구성으로 더 나은 결과를 보여줍니다. Babylon.js는 PBR(Physically Based Rendering) 파이프라인이 더 정교하지만, 기본 설정만으로는 그 장점이 드러나지 않습니다.

프레임 안정성 테스트:
같은 인스턴스를 다수 소환하여 프레임 안정화를 테스트한 결과, Babylon.js가 우위를 보였습니다. Babylon.js는 많은 객체를 다룰 때 프레임 드롭이 적고 안정적입니다. 고성능 퀄리티의 복잡한 씬에서 Babylon.js의 내부 최적화가 빛을 발합니다.

핵심 발견:
같은 3D 엔진이라고 같은 것이 아닙니다. 비슷할 뿐이지, 내부 로직에 따른 차이가 엄청났습니다. 라이팅 시각화 로직이 다르기 때문에 같은 모델을 로드해도 기본 구성으로 Three.js가 나은 것이고, 같은 인스턴스를 소환하여 프레임 안정화를 테스트한 결과가 다른 것도 각 엔진의 내부 렌더링 및 최적화 로직이 다르기 때문입니다.

적재적소 결론:
빠른 프로토타이핑이나 간단한 3D 뷰어, 제품 쇼케이스, 소규모 씬에는 Three.js가 적합합니다. 대규모 씬이나 많은 객체, 고성능 그래픽, 복잡한 렌더링, 게임 등 인터랙티브 콘텐츠에는 Babylon.js가 적합합니다. 이 테스트로 각 엔진의 장점이 명확해지고 프로젝트 성격에 따라 적용할 엔진을 판단할 수 있게 되었습니다.


제3장. Orillusion — WebGPU 차세대 엔진 테스트

Orillusion 소개:
중국에서 개발한 WebGPU 전용 3D 엔진입니다. 출시 약 2~3년이 되었으며 현재 v0.8.5-dev.9 알파 단계입니다. WebGL이 아닌 WebGPU만 사용하는 차세대 엔진으로, 차세대 엔진으로 자리잡을 수 있을지 테스트해보았습니다.

로딩 타임 비교:
Three.js 평균 2,369ms (기준 1배)
Babylon.js 평균 3,616ms (1.5배)
Orillusion 평균 7,183ms (3배)

Orillusion이 가장 느린 이유:
첫째, WebGPU 초기화 오버헤드입니다. WebGL은 getContext('webgl')로 동기적으로 즉시 컨텍스트를 생성하지만, WebGPU는 requestAdapter, requestDevice를 비동기로 호출하고, 렌더링 파이프라인도 명시적으로 컴파일해야 합니다. 이 과정에서 200~400ms가 추가됩니다.

둘째, 엔진 미성숙입니다. GLB 파서에 버그가 있어 수동 패치가 필요했고, 일부 노드 애니메이션이 지원되지 않는 등 알파 수준의 완성도입니다. GLTF_NODE_INDEX_PROPERTY가 undefined로 설정되는 프로퍼티 저장 버그도 발견되었습니다.

셋째, 최적화 부족입니다. Vite 프리번들링 버그를 우회하기 위해 2.16MB 비압축 소스를 사용해야 했습니다. Three.js와 Babylon.js는 최적화된 번들을 사용하는 것과 대비됩니다.

퀄리티 평가:
퀄리티는 오랜 경력을 가진 Three.js와 Babylon.js가 현재로서는 더 나은 것으로 판단됩니다.

그럼에도 주목할 점:
현재 Babylon.js가 프레임 안정성과 고성능 그래픽을 다루는 엔진으로 자리잡은 지금, 그 자리를 위협할 만한 엔진이 맞습니다. WebGPU로 고성능 연산 및 병렬 처리가 뛰어나 보입니다.

WebGPU가 빛나는 영역은 단일 모델 로드가 아닌 대규모 렌더링입니다. 100개 이상의 객체 렌더링에서 효율적이고, 1000개 이상의 드로우콜에서 CPU 오버헤드를 제거하며, GPU 컴퓨트 셰이더를 네이티브로 지원하고, 100만 개 이상의 파티클 시뮬레이션이 가능합니다.

결론:
현재는 프로덕션 투입에 이르지만, R&D 및 실험 단계로는 적합합니다. WebGPU가 표준화되면 고성능 연산이 필요한 영역에서 강력한 대안이 될 가능성이 높으며, Babylon.js의 고성능 영역을 대체할 잠재력을 보유하고 있어 주시가 필요합니다.


제4장. WebLLM — 브라우저에서 LLM 실행

WebLLM 소개:
웹 브라우저에서 WebGPU를 이용하여 LLM(대규모 언어 모델)을 로컬에서 실행하는 기술입니다. 서버 없이 브라우저의 GPU만으로 AI 추론을 수행합니다. https://webllm.mlc.ai/ 에서 JavaScript 라이브러리와 WebGPU용으로 변환된 모델 파일을 제공합니다.

동작 원리:
모델 제작사(Meta, Alibaba 등)가 Llama, Qwen 등 오픈소스 모델을 서버에서 대규모로 학습합니다. MLC-AI 팀이 학습 완료된 모델을 WebGPU에서 실행할 수 있도록 변환하고 JavaScript 라이브러리를 제공합니다. 개발자는 npm 패키지로 가져다가 브라우저에서 추론을 실행합니다.

중요한 점은 WebLLM은 추론(inference) 전용이라는 것입니다. 학습(training)은 불가능합니다. 브라우저에서 학습이 안 되는 이유는 VRAM과 연산량이 턱없이 부족하기 때문입니다.

또한 GPT(OpenAI), Gemini(Google), Claude(Anthropic)처럼 모델 가중치를 공개하지 않는 비공개 모델은 사용할 수 없습니다. Qwen(Alibaba), Llama(Meta) 같은 오픈소스 모델만 사용 가능합니다. GPU가 아무리 좋아도 다운로드할 가중치 파일 자체가 없으면 로컬에서 실행할 수 없습니다.

테스트 결과:
모델 다운로드 속도는 많이 느리지 않았습니다. 다시 웹에 접속 시 캐싱된 데이터를 불러와 초기 로드보다 빨랐습니다. GPT나 Gemini에 비교할 바는 안 되지만, 웹에서 WebGPU를 이용해 LLM 모델을 실행할 수 있다는 것에 의의를 둡니다.

활용 예시 프로젝트 — 3D AI Scene:
WebGPU와 Three.js를 결합한 예시 프로젝트를 만들었습니다. 사용자가 자연어로 "빨간 공을 만들어"라고 입력하면, WebLLM이 시스템 프롬프트의 규칙에 따라 이를 JSON 명령으로 변환합니다. 예를 들어 action은 create, shape는 sphere, color는 #ff0000 형태의 JSON을 생성합니다. 그러면 Three.js가 이 JSON 명령을 받아 3D 씬에 실제 오브젝트를 생성합니다.

여기서 LLM이 어떤 커맨드를 써야 할지 아는 것은 학습이 아니라 프롬프트 엔지니어링입니다. 시스템 프롬프트에 사용 가능한 액션 목록(create, delete, move, color, animate, light, camera, clear)과 파라미터를 테이블로 명시하고, 입출력 예시를 여러 개 제공하면, LLM이 이 규칙에 따라 자연어를 정해진 JSON 포맷으로 변환하는 번역기 역할을 합니다.

결론:
웹에서 WebGPU를 이용해 LLM 모델을 실행할 수 있다는 것 자체가 의미 있는 기술입니다. 프라이버시가 중요한 환경에서 데이터가 서버로 나가지 않는 장점이 있고, 모델 캐싱 후 오프라인 지원도 가능합니다. 3D와 AI의 결합 가능성을 확인한 것도 의미 있는 성과입니다.


종합 결론

첫째, 개발 구조 분리는 성능 손실 없이 가능합니다. 열악한 환경에서 차이가 있으나, 팀 독립성 확보에 유의미합니다.

둘째, Three.js와 Babylon.js는 각자의 장점이 명확합니다. Three.js는 빠르고 기본 퀄리티가 우수하며, Babylon.js는 대규모 씬과 프레임 안정성이 우수합니다. 같은 3D 엔진이라도 내부 로직이 다르기 때문에 결과가 크게 다르며, 프로젝트 특성에 맞는 엔진 선택이 중요합니다.

셋째, Orillusion은 현재는 미성숙하나 WebGPU 기반 고성능 연산에서 잠재력이 확인되었습니다. Babylon.js의 고성능 영역을 위협할 수 있는 차세대 엔진으로 주시가 필요합니다.

넷째, WebLLM을 통해 브라우저 로컬에서 LLM 실행이 가능함을 확인했으며, Three.js와 결합한 3D AI Scene 예시 프로젝트를 구현하였습니다.

전체적으로 WebGPU는 3D 렌더링뿐 아니라 AI 추론까지 브라우저에서 가능하게 하는 핵심 기술이며, 현재 WebGL 기반 생태계와 공존하면서 점진적으로 고성능 영역을 확장해 나갈 것으로 전망됩니다.
