### web-host(분리) vs combined(통합) 차이점

두 모드 모두 **iframe을 사용하지 않는다**. 동일 JS 컨텍스트에서 실행된다.

| 항목 | web-host (분리) | combined (통합) |
|------|----------------|----------------|
| 위젯 로드 방식 | 런타임 cross-origin `import()` (별도 서버) | 정적 import (같은 번들) |
| 필요 서버 수 | 2개 (host + widget) | 1개 |
| JS 번들 크기 | 호스트: 작음 / 위젯: 별도 | 모두 포함된 단일 번들 |
| GLB 로드 경로 | 위젯 서버 (:5101/:5201) → LAN → 클라이언트 | 통합 서버 (:5102/:5202) → LAN → 클라이언트 |
| 추가 오버헤드 | embed.js fetch + CORS | 없음 |
| 타임라인 스텝 | 6단계 (원격 모듈 fetch 포함) | 4단계 |

### 로딩 타임 측정 기준

```
시작: window.__PAGE_START = performance.now()  (HTML <script>에서 기록)
종료: mount().ready Promise 해결 시점
  = GLB 로드 완료

표시: performance.now() - __PAGE_START (ms)
```

## 3. 분석

### 3.1 왜 WebGPU(Orillusion)가 WebGL(Babylon/Three)보다 느린가?

측정 대상이 **"렌더링 FPS"가 아니라 "초기화+로딩 시간"**이라는 점이 핵심이다.

#### (1) WebGPU 초기화 오버헤드

WebGPU는 WebGL보다 초기화 과정이 본질적으로 더 무겁다.

| 단계 | WebGL | WebGPU |
|------|-------|--------|
| 컨텍스트 생성 | `canvas.getContext('webgl')` → 동기, 즉시 | `navigator.gpu.requestAdapter()` → 비동기 |
| 디바이스 획득 | 컨텍스트에 포함 | `adapter.requestDevice()` → 비동기 |
| 셰이더 컴파일 | 첫 drawcall 시 암묵적 컴파일 | 파이프라인 생성 시 **명시적 사전 컴파일** (WGSL→바이너리) |
| 파이프라인 | 암묵적 (GL state machine) | `createRenderPipeline()` 명시적 생성 |

WebGPU는 "런타임 끊김(jank) 없이 렌더링"하기 위해 초기화 시 셰이더/파이프라인을 미리 컴파일한다.
이것이 **초기 로딩 시간 증가**의 대가이며, 대신 **렌더링 중 프레임 드롭이 적다**는 장점이 된다.

#### (2) Orillusion 엔진의 미성숙

| 항목 | Babylon.js | Three.js | Orillusion |
|------|-----------|----------|-----------|
| 버전 | 7.x (안정) | r160+ (안정) | **0.8.5-dev.9 (알파)** |
| 개발 기간 | ~10년 | ~12년 | ~3년 |
| GLB 파서 | 프로덕션 검증됨 | 프로덕션 검증됨 | **버그로 인해 수동 패치 필요** |
| 최적화 수준 | LOD, 인스턴싱, 텍스처 스트리밍 | LOD, 인스턴싱 | 기본적 |

#### (3) 비압축 소스 파일 사용

Orillusion의 Vite prebundling 버그 우회를 위해 `orillusion.es.max.js` (~2.16MB)를 비압축 상태로 `src/lib/`에 복사해서 사용 중이다. Babylon.js와 Three.js는 최적화/tree-shaking된 번들을 사용한다.

| 엔진 | JS 파싱 대상 | 비고 |
|------|-------------|------|
| Babylon.js | 최적화된 번들 | Vite가 정상 prebundle |
| Three.js | 최적화된 번들 | Vite가 정상 prebundle |
| Orillusion | **2.16MB 비압축 소스** | prebundling 우회 |

LAN 환경에서 2.16MB 소스 전송 + 브라우저의 JS 파싱 시간이 추가된다.

#### (4) 추가 작업

Orillusion 앱은 Babylon/Three.js 테스트와 달리 추가 작업을 수행한다:
- HDR 큐브맵 로드 + SkyRenderer 구성
- Bloom + TAA 포스트 프로세싱
- 커스텀 노드 애니메이션 파서 (`glbNodeAnimation.ts`) — GLB를 **한 번 더 fetch+파싱**

#### (5) WebGPU의 장점이 빛나는 영역

WebGPU의 설계 목적은 **"초기화 속도"가 아니라 "대규모 렌더링 효율"**이다.

| 시나리오 | WebGL | WebGPU |
|---------|-------|--------|
| 단일 모델 로딩 속도 | 빠름 | 느림 (초기화 오버헤드) |
| 100개 오브젝트 렌더링 FPS | CPU 바운드 시작 | 여유 있음 |
| 1000+ 드로우콜 | CPU 병목 심각 | **CPU 오버헤드 대폭 감소** |
| GPU Compute 활용 | 불가 | **네이티브 지원** |
| 인스턴싱/간접 드로우 | 제한적 | 효율적 |

현재 테스트는 **GLB 1개 로드 후 안정화까지의 시간**을 측정하므로, WebGPU의 장점이 드러나지 않는 영역이다.

### 3.2 web-host(분리)와 combined(통합)의 차이가 왜 작은가?

#### LAN 환경 특성

이 테스트는 localhost가 아닌 **LAN 네트워크**를 경유한다.

```
Gram 노트북 ──── Wi-Fi/Ethernet ────► Ubuntu 서버 (192.168.0.16)
                  RTT: ~1-5ms
                  대역폭: ~100Mbps-1Gbps
```

LAN이라도 localhost와 비교하면 실제 네트워크를 통한 전송이므로:
- **27MB GLB 전송 시간이 의미 있음** (100Mbps 기준 ~2.2초, 1Gbps 기준 ~0.2초)
- embed.js fetch에 실제 RTT가 발생

#### 로딩 시간 구성 분석 (추정)

```
┌──────────────────────────────────────────────────────────┐
│                   전체 로딩 시간 구성                       │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  [1] 호스트 JS 로드+파싱          ~100-200ms             │
│  [2] embed.js fetch (web-host만) ~50-150ms  ← 차이 원인 │
│  [3] 3D 엔진 초기화               ~200-500ms             │
│  [4] GLB 27MB 전송 (LAN)         ~500-2500ms ← 최대 병목 │
│  [5] GLB 파싱 + GPU 업로드        ~300-800ms             │
│  [6] 셰이더 컴파일                ~100-300ms             │
│  [7] 8프레임 안정화 대기           ~130-270ms             │
│                                                          │
│  총합: ~1,380ms ~ 4,720ms                                │
│                                                          │
│  embed.js fetch 비중: 전체의 ~3-5%                       │
└──────────────────────────────────────────────────────────┘
```

**[4] GLB 네트워크 전송이 전체의 40-60%를 차지**하므로, [2]의 embed.js fetch (~50-150ms)는 전체 대비 미미하다.

#### 왜 Babylon은 web-host가 오히려 빠른가?

Babylon.js 결과에서 web-host(3,558ms)가 combined(3,616ms)보다 58ms 빨랐다.

가능한 원인:
1. **번들 크기 차이**: combined는 호스트+Babylon.js+위젯이 모두 한 번들이므로 **초기 JS 다운로드가 더 큼**
2. **병렬 로딩 효과**: web-host는 호스트 JS(작음)를 먼저 파싱하고, embed.js를 별도 요청으로 가져온다. 브라우저가 호스트 UI를 먼저 처리하는 동안 모듈 fetch가 병렬로 진행될 수 있다
3. **Docker 컨테이너 부하**: combined는 하나의 Vite 프로세스가 더 큰 번들을 서빙해야 하므로 미세하게 느릴 수 있다

#### 분리(widget) 방식이 더 빠를 수 있는 조건

| 조건 | 설명 |
|------|------|
| **코드 분할 효과** | 호스트 JS가 작아 브라우저가 빠르게 파싱 → 즉시 embed.js 요청 시작 |
| **HTTP/2 멀티플렉싱** | 여러 작은 리소스를 병렬 전송할 때 유리 |
| **캐시 재사용** | embed.js가 이미 캐시에 있으면 (이전 방문) 네트워크 요청 생략 |
| **CDN 엣지 캐시** | 프로덕션에서 widget을 CDN에 배포하면 가까운 엣지에서 서빙 |

현재 LAN 테스트에서도 **Babylon.js는 실제로 web-host가 58ms 빨랐다**. 이는 번들 분할에 의한 **병렬 로딩 효과**가 embed.js fetch 오버헤드를 상쇄한 것으로 보인다.

Three.js에서 combined가 빠른 이유는 Three.js의 번들 크기가 Babylon.js보다 작아서 통합 번들의 불이익이 적고, 대신 embed.js fetch의 오버헤드가 상대적으로 더 크게 작용한 것으로 추정된다.

#### 핵심 결론

> **분리(widget) 방식은 "극적으로 빠르거나 느리지" 않다.**
>
> - 병목이 **27MB GLB 전송**이기 때문에, embed.js를 분리하든 통합하든 전체 시간의 3-5%밖에 차이가 나지 않는다
> - 다만, **분리 방식은 번들 분할/병렬 로딩으로 인해 통합 방식과 동등하거나 약간 빠를 수 있다** (Babylon 결과가 이를 증명)
> - LAN이 아닌 **실제 인터넷 환경**에서는 RTT가 커지므로 분리 방식의 추가 요청 오버헤드가 더 두드러질 수 있다
> - 반대로 **CDN + 캐시 환경**에서는 분리 방식이 더 유리해질 수 있다

## 4. WebGPU가 유리한 테스트를 하려면

현재 테스트는 "초기 로딩 시간"을 측정하므로 WebGPU의 장점이 드러나지 않는다.
WebGPU의 장점을 확인하려면 다음과 같은 테스트가 필요하다:

### 제안 테스트

| 테스트 | 측정 지표 | WebGPU 이점 |
|-------|----------|------------|
| 인스턴스 100~1000개 추가 후 FPS | 평균 FPS | 드로우콜 CPU 오버헤드 감소 |
| 동적 오브젝트 대량 생성/삭제 | FPS 저하율 | GPU 커맨드 배칭 효율 |
| 복잡한 포스트 프로세싱 체인 | FPS | Compute Shader 활용 |
| 파티클 100만 개 시뮬레이션 | FPS | GPU Compute 네이티브 |

### 현재 Orillusion 앱에서 테스트하는 법

1. Add 버튼으로 인스턴스를 50, 100, 200개 순서로 추가
2. 각 단계에서 FPS 패널의 값을 기록
3. 동일 모델/동일 개수로 Babylon.js/Three.js와 비교
4. **드로우콜 수가 1000을 넘어갈 때부터** WebGPU의 CPU 절감 효과가 드러남

## 6. 로딩 시간 편차 분석 (같은 조건에서 3초 vs 8초)

### 현상

동일한 조건으로 반복 테스트해도 가끔 로딩 시간이 크게 튀는 경우가 있다.

- **테스트 방식**: Chrome 시크릿 모드 + 강력 새로고침 (Ctrl+Shift+R)
- **평소**: ~3초
- **가끔**: ~8초 (같은 날, 비슷한 시간대)

### 시크릿 + 강력 새로고침의 의미

| 조건 | 효과 |
|------|------|
| 시크릿 모드 | 이전 세션의 disk cache 없음, 쿠키/스토리지 초기화 |
| 강력 새로고침 | `Cache-Control: no-cache` 헤더로 요청, 브라우저 캐시 무시 |

즉 **매 테스트마다 27MB GLB를 네트워크로 전부 새로 다운로드**한다. 캐시 변수는 완전히 제거된 상태.

### 원인 분석

캐시가 제거되었으므로 범인 후보가 좁아진다.

#### 1순위: Wi-Fi 대역폭 변동 (가장 유력)

매번 27MB를 네트워크로 풀 다운로드하므로 Wi-Fi 상태가 로딩 시간을 **직접 결정**한다.

```
평소:  Wi-Fi 300Mbps → GLB ~0.7초 → 총 ~3초
가끔:  Wi-Fi  50Mbps → GLB ~4.3초 → 총 ~8초
                        ↑
                    이 차이만으로 5초 벌어짐
```

Wi-Fi는 같은 자리에서도 순간적으로 대역폭이 변동한다:

| 간섭 원인 | 영향 |
|----------|------|
| 같은 채널의 옆집 공유기 활성화 | 대역폭 반감 가능 |
| 같은 네트워크의 다른 기기가 대용량 다운로드 | 공유 대역폭 감소 |
| 블루투스 기기 간섭 (2.4GHz 대역) | 패킷 재전송 증가 |
| AP와 노트북 사이에 사람이 지나감 | 순간 신호 감쇠 |
| 전자레인지 등 전자파 간섭 | 2.4GHz 대역 마비 |

#### 2순위: 노트북 CPU 서멀 스로틀링

LG Gram은 얇은 노트북이라 열 관리가 타이트하다.

```
정상 상태:  CPU 4.0GHz → JS 파싱 + GLB 디코딩 빠름 → 3초
과열 상태:  CPU 1.8GHz → 처리 속도 절반            → 6-8초
```

직전에 무거운 작업(다른 3D 테스트, 영상 재생 등)을 했다면 CPU에 열이 쌓여 클럭이 낮아진다.
같은 시간대라도 **직전에 뭘 했느냐**에 따라 달라진다.

#### 3순위: Docker 컨테이너 리소스 경합

Docker 안에서 여러 서비스가 동시에 실행 중이다.

```
[full-babylon]         ─┐
[webhost-only-babylon]  ─┤ 같은 호스트 CPU / 메모리 / 디스크 I/O 공유
[full-three]           ─┤
[webhost-only-three]   ─┘
```

다른 컨테이너가 동시에 요청을 처리하거나 Node.js GC가 돌면 디스크 I/O / CPU 경합이 발생할 수 있다.

#### 4순위: V8 가비지 컬렉션 (GC)

27MB GLB를 `ArrayBuffer`로 받아서 파싱하면 수백MB의 임시 객체가 생긴다.
V8이 Major GC(전체 정리)를 실행하면 **수백ms 멈출 수 있다**.
다만 시크릿 모드는 매번 깨끗한 힙에서 시작하므로 영향은 상대적으로 적다.

### 범인 확인 방법

느릴 때 Chrome DevTools에서 확인:

| 확인 대상 | 방법 | 판단 |
|----------|------|------|
| **네트워크** | Network 탭 → GLB 파일의 **Time** 열 | 평소 800ms인데 5000ms면 → **Wi-Fi 문제** |
| **CPU 스로틀** | 작업관리자 → CPU 주파수 확인 | 기본 클럭의 절반 이하면 → **서멀 스로틀링** |
| **GC 멈춤** | Performance 탭 → 녹화 → 노란 "GC" 블록 | GC 블록이 500ms+ 이면 → **GC 문제** |

**가장 빠른 확인법**: 느릴 때 Network 탭에서 `free_cyberpunk_hovercar.glb`의 Time만 비교하면 된다.
- GLB Time이 길면 → 네트워크 문제
- GLB Time은 비슷한데 전체가 느리면 → CPU/GC 문제

### 편차를 줄이는 방법

| 방법 | 효과 | 난이도 |
|------|------|-------|
| **유선 LAN으로 테스트** | Wi-Fi 변동 완전 제거, 가장 확실 | 낮음 |
| 서버에서 GLB를 gzip/brotli 압축 서빙 | 27MB → ~5-8MB 전송량, 네트워크 영향 감소 | 중간 |
| 테스트 전 노트북 5분 방치 (쿨다운) | CPU 서멀 스로틀링 방지 | 낮음 |
| 불필요한 Docker 컨테이너 중지 | 서버 리소스 경합 감소 | 낮음 |
| 5회 이상 측정 후 최고/최저 제외한 평균 | 통계적 신뢰도 향상 | 낮음 |
